# 변경대비표 자동화 시스템 설계 문서

> [!summary]+ 한 줄 요약
> 임상시험 프로토콜 DOCX 두 버전을 넣으면, **변경대비표(Comparison Table of Change) DOCX**가 자동으로 나오는 시스템.

---

## 1. 이 시스템은 왜 만드는가

### 현재 문제

임상시험 프로토콜이 버전 업데이트될 때마다 **변경대비표**를 수작업으로 만들고 있습니다.

- 100페이지 문서를 한 줄씩 대조
- 텍스트뿐 아니라 표, 이미지, 머리글/바닥글까지 비교
- 변경 하나를 놓치면 규제 이슈로 직결
- 매번 수 시간~수일 소요

### 목표

```
V1.docx + V2.docx  →  [자동화 시스템]  →  변경대비표.docx
```

"**변경대비표 만들어줘**" 한마디로 끝나는 Claude Code Skill을 만드는 것이 최종 목표입니다.

---

## 3. 각 Phase 상세 설명

### Phase 1: 섹션 분할 (순수 Python)

> **핵심 질문**: "100페이지 문서를 어떻게 비교 가능한 단위로 나누는가?"

#### 왜 먼저 나눠야 하나

- 100페이지를 한 번에 비교하면 작은 변경이 묻힘
- 표, 그림 변경을 놓치기 쉬움
- 모델에 너무 많은 내용을 보내면 비용·정확도 모두 나빠짐

#### 어떻게 나누나

**Adaptive Heading 분할**:
1. DOCX의 XML 레벨에서 `w:body` 직계 자식을 순회 (python-docx API가 아닌 lxml 직접 사용)
2. Heading 1/2/3 기준으로 1차 분할
3. 30 blocks 초과 섹션만 Heading 4/5로 추가 분할
4. Table of Contents, 서명 페이지, 빈 Heading은 제외

> [!important] 핵심 발견: 자동 번호 복원
> 이 문서의 Heading 텍스트에는 번호가 없습니다 ("Introduction", "Study Plan" 등).
> 하지만 Word는 `numbering.xml`의 정의에 따라 "1. Introduction", "3. Study Plan"으로 렌더링합니다.
> 따라서 **numbering.xml을 파싱하여 번호를 복원**해야 변경대비표의 Item 열에 정확한 섹션 경로가 들어갑니다.

#### 산출물

```
work/
├── old/
│   ├── blocks.json           ← 문단+표 순서 보존된 블록 스트림
│   ├── section_index.json    ← 섹션 목록 (경로, 범위, 제외 여부)
│   ├── media_inventory.json  ← 이미지 SHA-256 해시
│   └── headers_footers.json  ← 머리글/바닥글 텍스트
├── new/
│   └── (동일 구조)
└── matched_pairs.json        ← old↔new 섹션 짝
```

#### 예시 문서 분석 결과

| 항목 | V1 | V2 |
|------|-----|-----|
| 총 문단 | 1,007 | 1,036 |
| 총 표 | 20 | 20 |
| 이미지 | 9 | 9 (3개 변경) |
| Heading 수 | 149 | 149 |
| Heading 매칭률 | — | **99%+** (131/132 일치) |

### Phase 2: 변경 후보 생성 (순수 Python)

> **핵심 질문**: "매칭된 섹션 짝에서 무엇이 바뀌었는가?"

#### 비교 대상 4가지

| 대상 | 도구 | 비교 방식 |
|------|------|----------|
| **텍스트** | diff-match-patch (Google) | 문단 단위 정렬 → character-level diff |
| **표** | 커스텀 로직 | 셀 단위 비교 (행/열 좌표, 병합 셀 정규화) |
| **이미지** | SHA-256 해시 | 바이너리 해시 비교 (같음/다름) |
| **머리글/바닥글** | diff-match-patch | 텍스트 비교 (버전, 날짜 등) |

#### 변경 유형 분류

- **add**: 새로 추가된 내용
- **delete**: 삭제된 내용
- **modify**: 수정된 내용 (character-level diff 포함)
- **move**: 같은 섹션 내에서 위치가 바뀐 문단

#### 텍스트 정규화

diff 전에 양쪽에 동일한 정규화 적용:
- 연속 공백/개행 → 단일 공백
- 불필요한 제어문자 제거
- 문단 끝 공백 제거

#### 산출물

```json
// work/diff/change_candidates.json (예시)
[
  {
    "section_path": "5. Treatments > 5.1. Identity of IP",
    "object_type": "text",
    "change_type": "modify",
    "location_hint": "paragraph_idx=17",
    "before": "Version 1.0",
    "after": "Version 2.0",
    "evidence": "@@ -1,11 +1,11 @@\n Version \n-1.0\n+2.0\n",
    "confidence": 1.0
  }
]
```

### Phase 3: Claude로 행 생성 (Claude Code CLI)

> **핵심 질문**: "기계가 찾은 변경을 사람이 읽기 좋게 정리하려면?"

이 단계에서 **처음으로 AI(Claude)가 등장**합니다.

#### Claude의 역할

Claude는 **변경을 찾는 것이 아니라, 이미 찾은 변경을 정리**합니다.

```
Phase 2가 찾은 것            Claude가 정리한 것
─────────────────            ──────────────────
before: "Version 1.0"   →   Previous Version: "Version 1.0"
after: "Version 2.0"    →   Current Version: "Version 2.0"
change_type: "modify"   →   Note: "프로토콜 버전 업데이트"
```

#### 기술 구현

- **Claude Code CLI**: `claude -p --output-format json --json-schema`
- **병렬 3개**: `ThreadPoolExecutor(max_workers=3)`으로 섹션별 동시 처리
- **재시도 2회**: JSON Schema 불일치 시 최대 2회 재시도
- **JSON Schema 강제**: 5열 양식(page, item, previous_version, current_version, note) 구조 보장
- **응답 파싱**: CLI 응답의 `structured_output` 필드에서 결과 추출 (`result` 아님)

#### 언어 감지

- 텍스트에 한글이 1글자라도 있으면 → **한국어** 변경대비표
- 한글이 전혀 없으면 → **영어** 변경대비표
- (한국 문서에는 영어가 혼용되지만, 영어 문서에는 한글이 없음)

#### Note 열 (변경 사유)

- 문서에 변경 사유가 명시되어 있지 않으므로 **Claude가 문맥 기반으로 추정**
- 확실하지 않은 사유는 "추정: ..." 형태로 표기

## 5. 주요 설계 결정과 이유

### "Claude가 직접 diff하지 않는 이유"

| 관점 | Claude에게 맡기면 | 로컬 스크립트로 하면 |
|------|-----------------|-------------------|
| **재현성** | 같은 입력에 다른 결과 가능 | 동일 입력 = 동일 결과 |
| **감사 가능성** | "왜 이걸 변경으로 잡았는지" 설명 불가 | 해시/좌표/diff patch로 증거 남김 |
| **비용** | 100페이지 전체를 토큰으로 전송 | 변경된 부분만 전송 |
| **정확도** | 표/이미지 변경을 놓칠 수 있음 | 셀 단위/해시 단위로 정밀 비교 |

→ **결론**: 로컬에서 찾고, Claude는 정리만.

### "왜 XML 레벨로 파싱하는가"

python-docx의 `doc.paragraphs` API는 **표를 순서대로 포함하지 않습니다**. 실제 문서에서는 "문단 → 표 → 문단 → 표" 순서로 콘텐츠가 섞여 있는데, 상위 API만 쓰면 표 20개가 순서 없이 별도 리스트로 나옵니다.

→ **해결**: `lxml`으로 `w:body` 직계 자식(`w:p`, `w:tbl`)을 직접 순회하여 **혼합 콘텐츠 순서를 보존**.

### "Heading 텍스트에 번호가 없는 문제"

```
XML Heading 텍스트: "Introduction"
Word 렌더링:       "1. Introduction"
TOC 항목:          "1.Introduction"
```

→ **해결**: `numbering.xml`에서 abstractNum 정의를 파싱하고, Heading 레벨별 카운터를 관리하여 번호를 복원. PoC 검증 완료.

### "왜 diff-match-patch인가"

| 엔진 | 장점 | 단점 |
|------|------|------|
| `difflib` (표준) | 설치 불필요, 사람 읽기 좋은 결과 | 최소 편집 보장 없음, 성능 이슈 |
| `diff-match-patch` (Google) | character-level, 빠르고 정밀 | 외부 패키지 (무료) |

→ 사용자가 **더 정밀한 쪽**을 선호하여 diff-match-patch 선택.

## 9. 향후 확장 가능성

- **퍼셉추얼 해시**: 이미지가 리사이즈/재압축만 된 경우 "유사" 판정
- **OCR**: 그림 속 텍스트 변경 감지 (Tesseract)
- **PDF 페이지 매핑**: DOCX→PDF 변환 후 정확한 페이지 번호 부여
- **Excel 출력**: 검토용 필터/정렬 기능이 필요한 경우
- **섹션 간 이동 감지**: 현재는 섹션 내 이동만 감지
