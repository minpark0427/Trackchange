## 2. 실행 방법

### 사전 준비

```bash
# 1) 프로젝트 폴더로 이동
cd /Users/min/Vibecoding/Trackchange

# 2) 의존성 설치 (최초 1회)
pip3 install -r requirements.txt
```

### 실행 명령어

```bash
python3 scripts/run_split.py \
  --old "SPONSOR_PROTOCOL001_Clinical Trial Protocol_V1.0_01Jul2019_Final_clean.docx" \
  --new "SPONSOR_PROTOCOL001_Clinical Trial Protocol_V2_02 Oct 2019_Clean.docx" \
  --out work/
```

| 옵션 | 설명 | 예시 |
|------|------|------|
| `--old` | 이전 버전 DOCX 파일 경로 | `"V1.docx"` |
| `--new` | 현재 버전 DOCX 파일 경로 | `"V2.docx"` |
| `--out` | 결과물 저장 폴더 | `work/` |

> **참고**: 파일명에 공백이 있으면 반드시 따옴표(`"..."`)로 감싸야 합니다.

## 3. 각 단계별 상세 설명

### Step 1~2: 블록 추출 (`extract_blocks.py`)

DOCX 파일의 XML을 직접 순회하여 문서를 **블록** 단위로 추출합니다.

**블록이란?** Word 문서의 최소 구성 단위입니다:

| 블록 유형 | 설명 | 예시 |
|-----------|------|------|
| `paragraph` | 텍스트 문단 (제목 포함) | "1.1. Unmet Medical Need" |
| `table` | 표 전체 (셀 텍스트 + 병합 정보 포함) | 임상시험 개요 표 |

**추출되는 정보**:
- 문단 텍스트, 스타일 (`Heading1`, `BodyText` 등)
- 제목 레벨 (Heading 1~5)
- **자동 번호 복원** — Word에서 표시되는 `1.`, `1.1.`, `5.2.3.` 등의 번호를 `numbering.xml`에서 파싱하여 재현
- 이미지 포함 여부
- 표의 셀 텍스트, 셀 병합 정보 (`gridSpan`, `vMerge`)

**산출물 예시** (`work/old/blocks.json` 중 일부):

```json
{
  "idx": 4,
  "type": "paragraph",
  "style": "BodyText",
  "text": "A Randomized, Double-blind, Placebo-controlled...",
  "has_image": false
}
```

```json
{
  "idx": 7,
  "type": "table",
  "rows": [
    { "cells": [
        { "text": "Investigational Product:", "gridSpan": 1, "vMerge": null },
        { "text": "COMPOUND_X", "gridSpan": 1, "vMerge": null }
    ]}
  ]
}
```

> V1, V2 각각에 대해 `blocks.json`, `media_inventory.json`(이미지 해시), `headers_footers.json`(머리글/바닥글)이 생성됩니다.

---

### Step 3~4: 섹션 분할 (`split_sections.py`)

블록 목록을 **섹션(Section)** 단위로 그룹화합니다.

#### 분할 기준

```
Heading 1  ─── 1차 분할선 ───────────────────────
  본문 블록들...
  Heading 2  ── 1차 분할선 ──────────────────────
    본문 블록들...
    Heading 3  ─ 1차 분할선 ─────────────────────
      본문 블록들...
      Heading 4  ─ 2차 분할선 (조건부) ──────────
        본문 블록들...
```

| 분할 단계 | 대상 | 조건 |
|-----------|------|------|
| **1차 분할** | Heading 1, 2, 3 | 항상 분할 |
| **2차 분할** (적응형) | Heading 4, 5 | 해당 섹션의 블록 수가 **30개 초과**일 때만 추가 분할 |

> **왜 적응형인가?** 프로토콜 문서는 섹션마다 길이가 크게 다릅니다. 짧은 섹션(30블록 이하)은 통째로, 긴 섹션은 하위 제목으로 추가 분할하여 이후 비교의 정밀도를 높입니다.

#### 자동 제외 항목

다음 섹션은 비교 대상에서 **자동 제외**(`excluded: true`)됩니다:

- Table of Contents (목차)
- List of Tables / Figures / Abbreviations
- Sponsor Protocol Signature Page
- Investigator Acknowledgement
- Clinical Study Synopsis
- 빈 제목 (텍스트 없음)

#### 산출물 예시 (`work/old/section_index.json` 중 일부):

```json
{
  "section_id": "sec_0010",
  "section_path": "1. Introduction > 1.1. Unmet Medical Need",
  "heading_level": 2,
  "heading_text": "Unmet Medical Need",
  "start_block_idx": 68,
  "end_block_idx": 75,
  "block_count": 7,
  "excluded": false,
  "has_tables": false,
  "has_images": false
}
```

| 필드 | 의미 |
|------|------|
| `section_id` | 고유 식별자 (`sec_0000` ~) |
| `section_path` | 계층 경로 (`>` 로 구분) |
| `heading_level` | 제목 레벨 (1~5) |
| `start_block_idx` ~ `end_block_idx` | `blocks.json`에서의 범위 |
| `block_count` | 섹션 내 블록 수 |
| `excluded` | 비교에서 제외 여부 |

---

### Step 5: 섹션 매칭 (`match_sections.py`)

V1과 V2의 섹션을 **1:1 매칭**합니다. 4단계 우선순위로 매칭:

```
우선순위 1: 정확 매칭     "Unmet Medical Need" = "Unmet Medical Need"     → exact
우선순위 2: 번호 매칭     "5.1 ..." ↔ "5.1 ..."                         → number
우선순위 3: 유사도 매칭   유사도 ≥ 80% (제목 텍스트 비교)                → fuzzy
우선순위 4: 미매칭        한쪽에만 존재하는 섹션                          → old_only / new_only
```

**산출물 예시** (`work/matched_pairs.json` 중 일부):

```json
{
  "match_type": "exact",
  "old_section_id": "sec_0009",
  "new_section_id": "sec_0009",
  "old_section_path": "1. Introduction",
  "new_section_path": "1. Introduction",
  "similarity": 1.0
}
```

> 실제 테스트 문서 기준: **총 128쌍** 매칭 (대부분 exact match)

## 4. 산출물 폴더 구조

실행 후 `work/` 폴더에 아래 파일들이 생성됩니다:

```
work/
├── old/
│   ├── blocks.json              ← V1 블록 (문단 + 표)
│   ├── section_index.json       ← V1 섹션 목록
│   ├── media_inventory.json     ← V1 이미지 해시
│   └── headers_footers.json     ← V1 머리글/바닥글
├── new/
│   ├── blocks.json              ← V2 블록
│   ├── section_index.json       ← V2 섹션 목록
│   ├── media_inventory.json     ← V2 이미지 해시
│   └── headers_footers.json     ← V2 머리글/바닥글
└── matched_pairs.json           ← V1↔V2 섹션 매칭 결과
```

## 5. 이후 파이프라인 (Phase 2~4)

문서 분할(Phase 1)은 전체 변경대비표 생성의 첫 단계입니다:

| Phase | 명령어 | 역할 |
|-------|--------|------|
| **Phase 1** (이 문서) | `run_split.py` | DOCX → 섹션 분할 + 매칭 |
| **Phase 2** | `run_diff.py` | 매칭된 섹션 간 텍스트/표/이미지 diff |
| **Phase 3** | `run_rows.py` | Claude AI로 변경대비표 행 생성 |
| **Phase 4** | `run_export.py` | 최종 변경대비표 DOCX 출력 |

전체 파이프라인 순차 실행:

```bash
python3 scripts/run_split.py  --old "V1.docx" --new "V2.docx" --out work/
python3 scripts/run_diff.py   --work-dir work/ --out work/diff/change_candidates.json
python3 scripts/run_rows.py   --work-dir work/ --out work/rows/change_rows.json
python3 scripts/run_export.py --work-dir work/ --out-dir work/output/
```

## 6. 자주 묻는 질문

### Q: 다른 프로토콜 문서에도 쓸 수 있나요?
네. Heading 1~5 스타일을 사용하는 표준 DOCX라면 모두 가능합니다. 특수 커스텀 스타일을 제목으로 쓰는 경우는 `split_sections.py` 수정이 필요할 수 있습니다.

### Q: 실행 중 에러가 나면?
각 Step에서 어느 단계에서 실패했는지 터미널에 표시됩니다. 가장 흔한 원인:
- 파일 경로 오타 (공백이 있는 파일명 따옴표 누락)
- `requirements.txt` 미설치 (`pip3 install -r requirements.txt`)
- 프로젝트 루트가 아닌 다른 폴더에서 실행

### Q: 원본 파일이 수정되나요?
**절대 아닙니다.** 파이프라인은 DOCX를 읽기만 하며, 모든 결과물은 `work/` 폴더에 생성됩니다.
