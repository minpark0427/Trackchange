좋아요.  
이번 작업은 **“큰 책 2권을 바로 비교하지 말고, 같은 챕터끼리 잘라서 비교하는 작업”**이라고 생각하면 이해가 쉽습니다.

## 전체 계획 한눈에 보기

### 1단계. 문서를 먼저 “작게 나눈다”

가장 중요합니다.  
100페이지짜리 문서를 한 번에 비교하면 너무 길어서, 작은 수정 하나도 놓치기 쉽습니다. 그래서 먼저 문서를 **섹션별로 잘라서 작은 묶음**으로 만들어야 합니다. DOCX는 내부적으로 XML과 미디어가 들어 있는 패키지 구조라서, 이런 식의 구조 기반 분리가 잘 맞습니다.

### 2단계. 나눈 조각끼리 짝을 맞춘다

예를 들어 old 문서의 `3.2 오류 처리`와 new 문서의 `3.2 오류 처리`를 한 쌍으로 만듭니다.  
이렇게 해야 같은 내용끼리 비교할 수 있습니다.

### 3단계. 조각별로 바뀐 부분만 찾는다

텍스트, 표, 그림을 따로따로 봅니다.  
문단은 문단끼리, 표는 표끼리, 이미지는 이미지끼리 비교합니다. 텍스트 diff는 `difflib` 같은 방식으로 사람이 읽기 좋은 차이를 만들 수 있습니다.

### 4단계. Claude Code는 “찾는 역할”보다 “정리하는 역할”로 쓴다

핵심은 Claude가 처음부터 모든 차이를 찾게 하지 않는 것입니다.  
로컬 스크립트가 먼저 바뀐 후보를 찾고, Claude Code는 그 후보를 받아서 **변경대비표 한 줄**로 정리하게 하는 편이 더 안정적입니다. Claude Code는 headless 실행과 JSON 구조 출력에 맞춰 쓰기 좋습니다.

### 5단계. 마지막에 전체 결과를 하나로 합친다

섹션별 결과를 모아서 CSV, XLSX, DOCX로 내보냅니다.  
여러 DOCX를 다시 합칠 때는 `docxcompose` 같은 도구를 쓸 수 있습니다.

---

# 가장 중요한 부분: 문서를 먼저 어떻게 나눌까

## 왜 먼저 나눠야 할까

중학생 수준으로 아주 쉽게 말하면:

- 100문제 시험지를 친구랑 답 비교할 때
    
- 처음부터 끝까지 한 줄씩 다 보면 너무 힘듭니다
    
- 그래서 보통 **1번~10번**, **11번~20번**처럼 끊어서 봅니다
    

문서도 똑같습니다.

한 번에 전체를 비교하면 생기는 문제는 이렇습니다.

- 너무 길어서 처리 속도가 느려짐
    
- 작은 오타 수정이 큰 변경에 묻힘
    
- 표나 그림 바뀐 것을 놓치기 쉬움
    
- 모델에 너무 많은 내용을 보내야 해서 비용과 보안 부담이 커짐
    

그래서 먼저 쪼개야 합니다.  
이게 이 작업의 **성공률을 올리는 첫 번째 기술**입니다.

---

## 문서를 나누는 기준은 무엇이 제일 좋을까

### 가장 좋은 기준: 제목(Heading) 단위

예를 들면 문서 안에 이런 구조가 있으면:

- 1. 개요
        
- 2. 설치 방법
        
- 2.1 사전 준비
    
- 2.2 설정
    
- 3. 오류 처리
        

이걸 기준으로 나누는 것입니다.

왜 좋냐면:

- 사람도 그렇게 읽음
    
- old와 new에서 같은 위치를 찾기 쉬움
    
- 비교 결과를 보고서에 적을 때도 이해하기 쉬움
    

즉,  
**“페이지 단위”보다 “제목 단위”가 더 좋습니다.**

Mammoth나 Pandoc 계열 도구도 이런 식으로 문서의 의미 구조를 다루는 데 도움이 될 수 있습니다. 다만 포맷 디테일이나 복잡한 표는 손실될 수 있어서, 보조 신호로 쓰는 게 좋습니다.

---

## 문서를 나누는 실제 원칙

### 원칙 1. Heading 1, 2, 3 기준으로 자른다

예:

- Heading 1 = 큰 장
    
- Heading 2 = 중간 항목
    
- Heading 3 = 세부 항목
    

이렇게 계층을 잡습니다.

예를 들면:

- `2 설치 방법`
    
- `2 > 2.1 사전 준비`
    
- `2 > 2.2 설정`
    

이런 식으로 **섹션 경로(section path)** 를 붙여 둡니다.  
나중에 변경대비표에 그대로 쓸 수 있습니다.

---

### 원칙 2. 너무 작은 조각으로는 나누지 않는다

너무 잘게 쪼개면 오히려 관리가 힘들어집니다.

예:

- 문장 하나마다 파일 분리 → 너무 많아서 실패
    
- 표 하나마다 무조건 따로 → 흐름이 깨짐
    

권장 방식은:

- 기본은 Heading 2 또는 Heading 3 단위
    
- 너무 긴 섹션만 추가로 더 쪼개기
    

즉,  
**적당히 의미 있는 크기**로 자르는 것이 중요합니다.

---

### 원칙 3. 제목이 없는 문서도 대비책이 있어야 한다

실무 문서는 제목 스타일이 엉망인 경우가 많습니다.

예를 들면:

- Heading 스타일을 안 썼다
    
- 그냥 굵게만 표시했다
    
- 번호는 있는데 스타일 정보가 없다
    

이 경우 대비책은 순서대로 이렇게 잡으면 됩니다.

#### 1순위: 진짜 Heading 스타일 사용

가장 정확합니다.

#### 2순위: 번호 패턴으로 추정

예:

- `1.`
    
- `1.1`
    
- `2.3.4`
    

이런 번호를 보고 제목처럼 추정합니다.

#### 3순위: 길이 제한으로 강제 분할

정말 구조가 엉망이면 마지막 수단으로:

- 문단 20개 단위
    
- 글자 수 3,000~5,000 단위
    
- 페이지 근사치 단위
    

로 나눕니다.

즉,  
**제목이 좋으면 제목으로, 없으면 번호로, 그것도 안 되면 길이로 자른다**  
이 순서로 가면 됩니다.

---

## 문서를 자를 때 꼭 같이 저장해야 하는 정보

문서를 나눈다고 해서 그냥 파일만 여러 개 만드는 게 아닙니다.  
각 조각에 **이름표**를 붙여야 합니다.

예를 들어 섹션 하나를 만들면 이런 정보가 같이 있어야 합니다.

- section_id: `sec_0032`
    
- section_path: `3.2 인증 > 오류 처리`
    
- heading_level: `2`
    
- start_block: 문서에서 시작 위치
    
- end_block: 문서에서 끝 위치
    
- source_doc: `old` 또는 `new`
    

이유는 간단합니다.

나중에 비교할 때  
“old의 sec_0032” 와 “new의 sec_0032” 를 연결해야 하기 때문입니다.

---

## “파일을 진짜로 나눌까?” vs “논리적으로만 나눌까?”

이건 아주 중요한 결정입니다.

### 방법 A. 진짜 DOCX 파일을 섹션별로 따로 만든다

예:

- `old_2_1.docx`
    
- `old_2_2.docx`
    
- `new_2_1.docx`
    
- `new_2_2.docx`
    

장점:

- 사람이 직접 열어 확인하기 쉬움
    
- 문제 생긴 섹션만 다시 돌리기 쉬움
    

단점:

- 파일 수가 많아짐
    
- 그림, 표, 머리글/바닥글 처리 귀찮음
    
- 다시 합칠 때 신경 쓸 게 많음
    

---

### 방법 B. 원본 DOCX는 그대로 두고, 내부적으로만 나눈다

예:

- 원본은 하나로 둠
    
- 대신 `blocks.json`, `section_index.json` 같은 구조 파일을 만듦
    

장점:

- 더 안정적임
    
- 원본 손상이 없음
    
- 자동화하기 좋음
    

단점:

- 사람이 바로 보기엔 덜 직관적임
    

---

### 추천

실무에서는 **B를 기본**으로 추천합니다.

즉,

- 원본 DOCX는 그대로 둔다
    
- 내부적으로 문단/표/이미지를 뽑아
    
- 섹션 인덱스를 만든다
    
- 필요할 때만 섹션별 미리보기 파일을 만든다
    

이 방식이 제일 덜 위험합니다.  
`python-docx`는 DOCX 생성/수정에는 좋지만, 문서 순서상의 혼합 콘텐츠를 다루려면 상위 API만으로는 한계가 있고, 표도 최상위 표만 바로 잡히는 부분이 있어 XML 레벨 순회가 더 안전할 수 있습니다.

---

# 중학생에게 설명하듯 단계별 실행 계획

## 1단계. 문서 두 개를 준비한다

- 이전 버전 문서(old)
    
- 새 버전 문서(new)
    

파일 이름도 단순하게 맞춥니다.

- `old.docx`
    
- `new.docx`
    

헷갈리지 않게 하는 게 중요합니다.

---

## 2단계. 문서를 “블록”으로 뜯어본다

문서를 바로 자르지 말고, 먼저 안에 뭐가 있는지 펼쳐봅니다.

블록 예시:

- 문단
    
- 표
    
- 그림
    
- 제목
    

쉽게 말하면,  
레고 상자를 열어서 블록 종류별로 보는 것입니다.

DOCX는 이런 구조 추출에 맞는 OOXML 패키지 형식입니다.

---

## 3단계. 제목을 기준으로 구역선을 긋는다

이 단계가 핵심입니다.

예:

- `1. 개요`부터 `2. 설치 방법` 전까지 = 섹션 1
    
- `2. 설치 방법`부터 `3. 오류 처리` 전까지 = 섹션 2
    
- 이런 식으로 문서 전체에 구역선을 긋습니다
    

즉,  
문서를 자르는 게 아니라 먼저 **“어디서부터 어디까지가 한 덩어리인지 표시”** 하는 것입니다.

이 결과를 `section_index.json`처럼 저장하면 좋습니다.

---

## 4단계. 각 섹션에 고유 이름을 붙인다

예:

- `sec_001 = 1. 개요`
    
- `sec_002 = 2. 설치 방법`
    
- `sec_003 = 2.1 사전 준비`
    

이제 old와 new 둘 다 같은 방식으로 이름표를 붙입니다.

그래야 나중에:

- old의 `sec_003`
    
- new의 `sec_003`
    

을 비교할 수 있습니다.

---

## 5단계. 너무 긴 섹션은 한 번 더 나눈다

예를 들어 `3. 오류 처리`가 15페이지라면 너무 깁니다.

이럴 땐:

- `3. 오류 처리 / part 1`
    
- `3. 오류 처리 / part 2`
    
- `3. 오류 처리 / part 3`
    

처럼 추가 분할합니다.

기준은 보통 이런 식이면 됩니다.

- 문단 수 기준
    
- 글자 수 기준
    
- 표 개수 포함 기준
    

중요한 건  
**너무 긴 덩어리를 모델에 그대로 던지지 않는 것**입니다.

---

## 6단계. old와 new 섹션을 짝지어 매칭한다

이제 같은 제목끼리 연결합니다.

예:

- old `2.1 사전 준비` ↔ new `2.1 사전 준비`
    
- old `3.2 오류 처리` ↔ new `3.2 오류 처리`
    

만약 제목이 조금 달라졌으면:

- 번호
    
- 제목 유사도
    
- 위치 순서
    

를 같이 봐서 짝지어야 합니다.

---

## 7단계. 비교는 로컬 스크립트가 먼저 한다

이 단계에서는 Claude를 아직 많이 쓰지 않습니다.

먼저 스크립트가:

- 문단 차이
    
- 표 셀 차이
    
- 이미지 해시 차이
    

를 찾아서 **변경 후보 목록**을 만듭니다.

이미지는 `word/media` 안의 파일 해시 비교로 “같은지/다른지”를 먼저 볼 수 있습니다.

---

## 8단계. Claude Code에게는 “정리”만 맡긴다

이제 Claude Code CLI로 각 섹션의 변경 후보를 넘깁니다.

Claude에게 시키는 일은 이런 것입니다.

- 이 변경은 추가/삭제/수정 중 무엇인가
    
- 사람이 읽기 쉽게 한 줄로 요약해라
    
- 변경대비표 형식의 JSON으로 답해라
    

Claude Code는 `-p`, `--bare`, `--output-format json`, `--json-schema` 같은 방식으로 자동화에 맞게 돌릴 수 있습니다.

---

## 9단계. 섹션별 결과를 모두 모은다

각 섹션에서 나온 결과를 한 파일로 모읍니다.

예:

- `change_rows_001.json`
    
- `change_rows_002.json`
    
- ...
    
- 최종 `report.xlsx`
    
- 최종 `report.docx`
    

---

## 10단계. 사람이 마지막 검토를 한다

자동화가 다 해도 마지막 검토는 꼭 필요합니다.

특히 봐야 할 것은:

- 표 셀 값이 정확히 잡혔는지
    
- 그림 교체가 진짜 변경인지
    
- 오타 수정과 의미 변경이 구분됐는지
    

---

# 실무용으로 정리한 “문서 분할 규칙”

## 추천 규칙

### 기본 규칙

- Heading 2 기준으로 1차 분할
    
- Heading 3 기준으로 2차 세분화
    
- 너무 긴 섹션만 추가 분할
    

### 예외 규칙

- Heading이 없으면 번호 패턴으로 분할
    
- 번호도 없으면 글자 수 또는 문단 수 기준으로 분할
    

### 안전 규칙

- 원본 DOCX는 절대 직접 잘라서 덮어쓰지 않기
    
- 분할 결과는 인덱스 파일로 관리하기
    
- 필요하면 미리보기용 DOCX만 별도 생성하기

# 가장 추천하는 운영 방식

## 가장 현실적인 플랜

### 문서 분할 전략

- 원본 DOCX는 그대로 보관
    
- 파이썬으로 문단/표/이미지 블록 추출
    
- Heading 기준으로 `section_index` 생성
    
- 너무 긴 섹션만 part 단위로 추가 분할
    

### 비교 전략

- 로컬에서 먼저 차이 후보 생성
    
- Claude Code는 후보를 설명문으로 정리
    
- JSON으로 받고 최종 표로 변환
    

### 출력 전략

- 검토용 Excel
    
- 제출용 DOCX 변경대비표
    
- 필요하면 섹션별 비교 로그 보관

# 아주 짧게 요약

## 핵심만 말하면

이 작업의 성패는 **Claude를 잘 쓰는 것보다, 문서를 먼저 잘 나누는 것**에 달려 있습니다.

가장 좋은 방법은:

1. **제목(Heading) 기준으로 문서를 나눈다**
    
2. **너무 긴 섹션만 추가로 더 쪼갠다**
    
3. **원본은 건드리지 않고, 인덱스 파일로 관리한다**
    
4. **로컬 스크립트가 차이를 찾고, Claude Code는 설명만 정리한다**
