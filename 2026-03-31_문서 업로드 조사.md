# Claude Code 기반 대용량 DOCX 변경대비표 자동화 CLI 워크플로우 설계 리서치

## 경영진 요약

대용량 DOCX(약 100페이지) 버전 업데이트 시 “텍스트·표·그림(이미지)” 변경사항을 빠짐없이 식별해 **변경대비표(변경점 목록 + 근거/위치 + 전후 요약)**로 내보내는 일은, 단순 텍스트 diff만으로는 충분하지 않습니다. DOCX는 OOXML(Office Open XML) 기반의 **ZIP 패키지 + XML 파트 + 관계(relationships) + media(이미지)** 구조를 갖고 있으며, 시각적 레이아웃(페이지/줄바꿈)은 렌더러(Word/LibreOffice) 의존성이 크기 때문입니다. citeturn8search19turn8search15turn8search3turn8search11

본 리서치는 **“결정적(Deterministic) 구조 추출·선별(diff 후보 생성) → LLM(Claude)로 사람 읽기 좋은 변경 행(row) 생성 → 표·파일로 내보내기”**의 2단계 아키텍처를 핵심 권장안으로 제시합니다. 즉, Claude(Claude Code CLI)는 *Diff를 직접 계산*하기보다 **이미 변화 가능성이 높은 후보만 정리·서술·표준화**하는 데 쓰고, 변경 탐지는 가능한 한 로컬에서 재현 가능하게 수행합니다(정확도·감사 가능성·비용 측면에서 유리).

Claude Code CLI는 `claude -p`(print/headless 모드)와 `--bare`(자동 탐색 훅/스킬/플러그인/MCP/메모리 등을 스킵해 CI/스크립트 재현성 강화), `--output-format json` 및 `--json-schema`(종단에서 검증된 구조적 JSON 반환) 같은 기능을 제공하므로, 섹션 단위 결과를 **스키마 기반 JSON**으로 안정적으로 모아 변경대비표를 생성하는 CLI 파이프라인에 적합합니다. citeturn1view0turn15view0turn4view0

성능·비용 측면에서는 Anthropic API의 **Message Batches(비동기 배치 처리)**를 활용하면 대량 요청을 비용 효율적으로 처리할 수 있으나, 공식 문서상 배치 기능은 **ZDR(Zero Data Retention) 미적용**이므로 민감 문서에는 주의가 필요합니다. citeturn5view0turn10view2  
보안·프라이버시 측면에서는 Claude Code가 “프롬프트 및 모델 출력”을 네트워크로 전송한다는 점이 명시되어 있으며, 기업 환경에서는 Claude Code/Claude API의 ZDR 적용 조건을 검토한 뒤(가능하면 ZDR) **전송 최소화(변경 후보만)** 전략이 중요합니다. citeturn10view0turn10view1turn10view2

## 목표와 성공 기준

### 목표 정의

이 워크플로우의 목표는 두 DOCX 버전(예: v1, v2)을 입력받아 다음을 자동 생성하는 것입니다.

문서의 각 섹션(제목/Heading 기준)에서 **추가/삭제/수정/이동**된 항목을 텍스트·표·이미지 단위로 탐지하고, 사람이 검토 가능한 수준의 “변경대비표 행(row)”로 구조화하여 DOCX/CSV/Excel/JSON 등으로 출력합니다.

### 성공 기준

정확도 측면에서는 다음을 “실무적으로 납득 가능한 자동화 성공”으로 간주합니다.

변경된 문단/표/이미지를 높은 재현율로 찾아내고(“빠짐 최소화”), 각 변경에 대해 **(a) 위치(섹션 경로) (b) 전/후 요약 (c) 근거(텍스트 스니펫·표 좌표·이미지 해시/파일명 등)**를 제공한다. 특히 표·그림 변경은 “원본 대비 무엇이 바뀌었는지”가 명확히 서술되어야 한다.

운영 측면에서는 실행이 스크립트로 재현 가능하고, 오류/중단 시 재시도가 쉬우며, 모델 호출 비용이 통제 가능해야 합니다(`--max-budget-usd` 같은 예산 상한, 또는 요청 배치/스로틀링). citeturn1view0turn15view0

## 가정과 의사결정 포인트

요청에서 “OS/언어/모델/키/로컬 vs 클라우드 처리”가 미정이므로, 아래를 설계 가정으로 명확히 두고 선택 가능한 분기안을 함께 제시합니다.

### 실행 환경

운영체제는 Linux/macOS/Windows 중 어느 것이든 가능하도록, 파이프라인은 (1) **Python 기반 핵심 로직** + (2) 선택적으로 Bash/PowerShell 래퍼로 구성하는 것을 기본으로 합니다. LibreOffice, Pandoc, Tesseract 같은 도구는 OS별 설치 차이가 있으므로 “옵션 기능”으로 두고, 없을 때도 기본 diff가 돌아가게 설계합니다. LibreOffice의 `soffice --convert-to ...` 형태 CLI 사용은 공식 도움말에도 문서화되어 있습니다. citeturn6search10

### 프로그래밍 언어

권장 언어는 Python입니다. 이유는 다음 라이브러리 생태계가 성숙해 있기 때문입니다.

DOCX 읽기/쓰기: `python-docx`는 .docx 생성·수정 라이브러리로 공식 문서가 잘 정리되어 있고 citeturn8search8, 병합은 `docxcompose`(CLI 포함)로 보완할 수 있습니다. citeturn16view0  
Excel: `openpyxl`은 xlsx/xlsm 등 Excel OOXML 파일을 읽고 쓰는 라이브러리로 문서가 안정적입니다. citeturn7search3turn7search15

### Claude 사용 방식

Claude Code CLI는 여러 인증 방식을 지원합니다(Claude.ai, Teams/Enterprise, Claude Console, Bedrock/Vertex/Foundry 등). citeturn15view1  
이 파이프라인은 **“비대화형 실행(print/headless)”**을 적극 활용합니다. Claude Code 공식 문서에서 `-p`(print)와 `--bare`(재현성/속도용) 사용을 권장하며, `--output-format json` 및 `--json-schema`로 구조적 출력을 받을 수 있음을 안내합니다. citeturn15view0turn1view0turn4view0

모델/버전은 조직 정책과 비용/정확도 트레이드오프에 따라 달라질 수 있으므로, CLI의 `--model` 플래그(예: `sonnet`, `opus` alias 또는 full name)로 외부화하는 것을 기본으로 합니다. citeturn1view0

### 로컬 처리 vs 클라우드 처리

권장 기본값은 **로컬에서 구조 추출·diff 후보 생성**, Claude에는 **diff 후보와 최소 컨텍스트만 전송**하는 방식입니다(프라이버시·비용·속도에 유리). Claude Code는 프롬프트/출력 데이터를 네트워크로 전송한다고 명시되어 있으므로, 민감 문서에서는 특히 전송량 최소화가 중요합니다. citeturn10view0

데이터 보존/프라이버시는 요구 수준에 따라 다음 중 하나를 선택합니다.

Enterprise에서 Claude Code ZDR: Claude Code ZDR은 Enterprise 환경에서 “응답 반환 후 저장하지 않는다”는 식으로 설명되며, 일부 기능은 ZDR 하에서 비활성화될 수 있습니다. citeturn10view1  
Anthropic API ZDR: Messages API 등은 ZDR 적용 가능이며, ZDR 계약 하에서는 응답 후 저장되지 않음을 한국어 문서가 명시합니다. citeturn10view2  
주의: Message Batches는 “ZDR 대상 아님”이라고 공식 문서에 명시되어 있습니다(대량 처리에는 유리하지만 민감 문서엔 부적합할 수 있음). citeturn5view0turn10view2

## 도구·라이브러리 스택과 대안 비교

이 섹션은 “필요 기능 → 권장 도구 → 대안 → 트레이드오프” 관점으로 정리합니다.

### DOCX 파싱/정규화/분할 도구 비교

DOCX는 OOXML(ISO/ECMA 표준) 기반이며 패키징/표현/관계 구조를 포함합니다. citeturn8search19turn8search11turn8search15  
이 구조 때문에 “텍스트만” 뽑는 툴과 “문서 구조(제목/표/이미지)”를 살리는 툴은 결과물이 다릅니다.

| 접근 | 장점 | 단점/주의 | 적합한 경우 |
|---|---|---|---|
| `python-docx` | .docx 생성/수정까지 가능(변경대비표 DOCX 출력에 유리). citeturn8search8 | 상위 레벨 API로는 **문서 순서의 혼합 콘텐츠(문단+표+이미지) 처리**가 까다롭고, `Document.tables`는 “상위 레벨 표만” 포함하며 중첩 표/리비전 마크 표는 제외될 수 있음. citeturn8search5 | “변경대비표 DOCX 생성”과 “구조 기반 분할”을 하나의 스택으로 처리하고 싶을 때 |
| Open XML SDK(.NET) | Office 문서를 프로그래밍으로 생성/조작 가능(공식). citeturn8search3 | Python 생태계 대비 구현 난이도/러닝커브. CLI 파이프라인에서 .NET 런타임 의존 | .NET 조직 표준이 있는 엔터프라이즈 환경 |
| Pandoc(`docx → markdown/native`) | 다양한 포맷 변환 및 AST 기반 필터/파이프라인 구성 가능. “구조 요소 보존, 포맷 디테일은 손실 가능”을 문서가 명시. citeturn11view0 | 복잡한 표/특수 레이아웃은 손실 가능. citeturn11view0 | “제목 기반 섹션 분할”과 “텍스트 중심 diff” 최적화 |
| Mammoth(`docx → HTML`) | *의미적 정보 기반*으로 깨끗한 HTML 생성, 스타일 디테일은 의도적으로 무시. citeturn6search0turn6search4 | 복잡 문서에 완전 변환 보장 어려움(문서가 mismatch 언급). citeturn6search0 | 제목/문단 등 의미 구조를 안정적으로 추출해 섹션 단위 비교에 쓰고 싶을 때 |

### Diff 엔진 비교

텍스트 차이는 “사람이 이해하기 쉬운 형태”가 중요합니다. Python 표준 `difflib.SequenceMatcher`는 “최장 연속 일치 구간 + 재귀” 기반이며 **최소 편집(minimal edit) 보장은 없지만 사람이 보기 좋은 결과를 지향**한다고 문서가 명시합니다. citeturn13view0

| 엔진 | 장점 | 단점/주의 | 적합한 경우 |
|---|---|---|---|
| `difflib`(표준) | 표준 라이브러리, 사람이 보기 좋은 결과 지향. citeturn13view0 | 최악 시간복잡도/특정 케이스 성능 이슈 가능(문서에 시간 복잡도 언급). citeturn13view0 | “중간 규모” 섹션/문단 비교, 외부 의존 최소 |
| `diff-match-patch`(Google) | Myers diff + 전/후 클린업, 고성능 텍스트 diff/patch 라이브러리로 설명. citeturn7search1 | 라이브러리 생태계/유지보수 이슈는 패키지별 확인 필요(파이썬 포크/래퍼 존재). citeturn7search5 | 긴 텍스트에서 빠르고 일관된 diff 품질이 필요할 때 |
| Levenshtein/유사도 | 편집거리·유사도 계산이 빠른 C 확장 모듈 계열. citeturn7search2turn7search10 | “정렬/매칭”에는 좋지만, 실제 사람이 읽는 diff(삽입/삭제 하이라이트) 로직은 별도 설계 필요 | 문단 재배치/유사 문단 매칭(align) 보조 지표 |

### 이미지/그림 변경 감지 접근 비교

DOCX 내부 이미지(또는 도형/차트의 결과물)는 “바이너리/렌더링” 차원이 섞이기 때문에, 최소 2단계로 나누는 것을 권장합니다.

| 접근 | 아이디어 | 장점 | 한계 |
|---|---|---|---|
| 바이너리 해시(SHA-256) | `word/media/*` 바이너리의 SHA-256 비교 | 구현이 단순, “같음/다름” 판정이 명확 | 메타데이터만 바뀌어도 다름. “거의 같은 그림” 판단 불가 |
| 퍼셉추얼 해시(pHash/dHash) | 이미지 재샘플링 기반 유사도(“비슷함”) | 그림의 사소한 압축/리사이즈 변화에서 “유사” 판정 가능 | 라이브러리 의존 + 임계값 튜닝 필요 |
| PDF 렌더링 비교 | DOCX를 PDF로 렌더링 후 페이지 이미지 비교 | “사용자가 보는 결과물” 기준 비교 가능 | 렌더러 차이/시간/정합성 비용 큼. LibreOffice 변환 필요 citeturn6search10 |
| OCR 기반 | 그림 안 텍스트를 OCR(Tesseract) 후 텍스트 diff | “그림 속 글자 변경” 탐지에 유리 | OCR 품질/언어팩/이미지 품질에 좌우. CLI 사용 문서 존재 citeturn6search3 |

## 권장 CLI 워크플로우 상세 설계

### 전체 파이프라인 개요

핵심은 “섹션 분할 + 구조 정규화 + 변경 후보 생성 + Claude로 행(row) 생성 + 산출물 생성”입니다.

```mermaid
flowchart TD
  A[old.docx] --> B[Extract 구조/자산<br/>blocks.json + media/]
  C[new.docx] --> D[Extract 구조/자산<br/>blocks.json + media/]

  B --> E[Section split/normalize<br/>section_index.json]
  D --> F[Section split/normalize<br/>section_index.json]

  E --> G[Deterministic diff<br/>change_candidates.json]
  F --> G

  G --> H{민감도/정책}
  H -->|ZDR 필요/전송 최소화| I[Claude Code CLI (-p --bare)<br/>변경 후보+컨텍스트만 전송]
  H -->|대량/비동기/저비용| J[Anthropic Message Batches API<br/>(주의: ZDR 미적용)]
  
  I --> K[change_rows.ndjson/json]
  J --> K

  K --> L[Aggregate/validate<br/>schema+dedupe+sort]
  L --> M[Export<br/>DOCX/CSV/XLSX/JSON]
```

### 파일/폴더 레이아웃(예시)

```
docx-change/
  inputs/
    old.docx
    new.docx
  work/
```

#### DOCX 구조 추출

DOCX는 OOXML 패키징 모델(OPC) 기반 ZIP 컨테이너이며, `word/document.xml`, `styles.xml`, `numbering.xml`, `word/media` 등이 핵심 구성요소로 알려져 있습니다. citeturn8search15turn8search7turn8search19  
따라서 추출 단계는 크게 두 가지 정보를 생성합니다.

텍스트/표 등 “블록”의 순서 있는 목록(block stream)  
이미지/미디어 자산 목록(media inventory) + 해시

`python-docx`는 문서 생성/수정에 강점이 있으나, 문서 내 표 목록(`Document.tables`)은 상위 레벨 표만 포함하고 중첩 표는 포함되지 않는다고 명시되어 있으므로, “정확한 순서+중첩 구조”가 필요하면 XML 레벨 순회(`w:p`, `w:tbl`)를 병행하는 편이 안전합니다. citeturn8search5turn8search13

아래는 “혼합 콘텐츠 스트림”을 만들기 위한 Python 스켈레톤(개념 코드)입니다.

```python
# scripts/extract_docx.py (개념 스켈레톤)
from __future__ import annotations
import hashlib, json, zipfile
from pathlib import Path
from docx import Document

def sha256_bytes(b: bytes) -> str:
    return hashlib.sha256(b).hexdigest()

def extract_media(docx_path: Path, out_media_dir: Path) -> dict:
    """word/media 아래 바이너리를 추출하고 sha256 인벤토리 생성."""
    inv = {}
    with zipfile.ZipFile(docx_path) as z:
        for name in z.namelist():
            if name.startswith("word/media/"):
                data = z.read(name)
                out_path = out_media_dir / Path(name).name
                out_path.write_bytes(data)
                inv[name] = {"file": str(out_path), "sha256": sha256_bytes(data), "bytes": len(data)}
    return inv

def extract_blocks(docx_path: Path) -> list[dict]:
    """python-docx로 문단 중심 추출(표/이미지는 확장 필요)."""
    doc = Document(str(docx_path))
    blocks = []
    for i, p in enumerate(doc.paragraphs):
        style = (p.style.name if p.style else "")
        text = p.text or ""
        blocks.append({"idx": i, "type": "paragraph", "style": style, "text": text})
    # NOTE: doc.tables는 상위 레벨만 포함될 수 있음. (중첩 표 제외) citeturn8search5
    return blocks

def main(docx_path: Path, out_dir: Path):
    out_dir.mkdir(parents=True, exist_ok=True)
    media_dir = out_dir / "media"
    media_dir.mkdir(exist_ok=True)

    media = extract_media(docx_path, media_dir)
    blocks = extract_blocks(docx_path)

    (out_dir / "media_inventory.json").write_text(json.dumps(media, ensure_ascii=False, indent=2), encoding="utf-8")
    (out_dir / "blocks.json").write_text(json.dumps(blocks, ensure_ascii=False, indent=2), encoding="utf-8")

if __name__ == "__main__":
    import argparse
    ap = argparse.ArgumentParser()
    ap.add_argument("--docx", required=True)
    ap.add_argument("--out", required=True)
    args = ap.parse_args()
    main(Path(args.docx), Path(args.out))
```

정규화는 diff 노이즈를 줄이는 핵심입니다(예: 공백/줄바꿈, 자동 번호, 날짜/필드 등). Pandoc 문서는 변환이 구조 요소는 보존하려고 하지만 포맷 디테일은 손실될 수 있고, 복잡한 표 등은 변환 손실이 있을 수 있다고 명시합니다. citeturn11view0  
따라서 정규화는 “문서 모델 손실”을 유발할 수 있는 변환을 기본값으로 삼지 말고, **동일한 규칙을 old/new에 똑같이 적용**하는 수준으로 제한합니다.

#### 변경 후보 생성(Deterministic diff)

변경 후보 생성은 LLM을 배제하고 로컬에서 수행합니다. 이유는 다음과 같습니다.

재현 가능성: 동일 입력이면 동일 후보가 나와야 한다.  
감사 가능성: “왜 이 변경이 잡혔는지”를 로그/해시로 남길 수 있다.  
비용 절감: LLM에는 “진짜 바뀐 것”만 보내면 된다.

텍스트 블록 diff는 `difflib` 또는 `diff-match-patch`를 선택할 수 있습니다. Python `difflib.SequenceMatcher`는 “최소 편집 보장은 없지만 사람이 보기 좋은 매칭을 제공”한다고 공식 문서에 명시되어 있습니다. citeturn13view0  
`diff-match-patch`는 Myers diff + 전/후 클린업으로 성능/품질을 개선한다고 설명됩니다. citeturn7search1

표 diff는 “행/열 단위 정규화 → 셀 단위 변경” 형태가 필요합니다. `python-docx`는 표 객체 API가 존재하나(행/셀 접근 등) citeturn0search2turn8search8, Word 표는 병합 셀/불규칙 그리드 등을 허용하므로 “정규화(행 길이 맞춤)”가 필요할 수 있다는 점이 API 문서에 언급되어 있습니다. citeturn0search2

이미지 diff는 **(1) media 바이너리 해시 비교(정확한 동일성) + (2) 선택적으로 퍼셉추얼 해시/렌더링/OCR(유사성·그림 속 텍스트)**를 조합합니다. OCR이 필요하면 Tesseract CLI 사용법이 공식 문서로 제공됩니다. citeturn6search3

### 페이지/섹션 참조를 어떻게 붙일 것인가

DOCX는 “페이지”가 고정 값이 아니라 렌더러/폴트/여백/프린터 설정에 따라 달라질 수 있어, 구조적으로 페이지 번호를 정확히 매기기 어렵습니다(이 부분은 OOXML 구조 특성상 일반적으로 알려진 한계이며, 본 워크플로우에서는 **섹션 경로**를 1차 식별자로 권장).

정말 페이지가 필요하면, DOCX를 PDF로 변환한 뒤(PDF는 페이지 개념이 고정) 텍스트 스니펫을 PDF에서 검색해 “해당 스니펫이 등장한 페이지”를 페이지 힌트로 부여할 수 있습니다. LibreOffice는 `soffice --convert-to 'pdf:writer_pdf_Export: {...}'` 형태로 페이지 범위를 포함한 PDF 변환 파라미터를 CLI에서 지원합니다. citeturn6search10  
다만 이 방법은 **(a) 변환 품질 (b) 동일한 렌더링 환경 확보 (c) 스니펫이 여러 페이지에 등장하는 경우** 등의 운영 난점이 있어, “옵션 기능”으로 두는 것이 현실적입니다.

### 산출물 DOCX 병합/재구성

섹션별로 분할된 DOCX를 다시 합쳐야 하는 요구가 있다면, `docxcompose`가 “docx 파일을 append/concatenate”하는 라이브러리이며 CLI 스크립트도 제공하고, “첫 문서의 헤더/푸터를 템플릿으로 사용하고 나머지는 무시”한다고 명시합니다. citeturn16view0  
이는 “섹션별 비교 후 최종 DOCX 합치기” 전략과 맞지만, 헤더/푸터 정책이 요구사항과 부합하는지 사전 검증이 필요합니다. citeturn16view0

### 성능·확장성

#### Claude 호출량을 줄이는 것이 최우선

Claude Code는 네트워크로 프롬프트/출력을 전송하므로 citeturn10view0, 토큰/호출 수를 줄이는 것이 비용·속도·프라이버시에 동시에 이득입니다. 따라서 다음을 권장합니다.

섹션 전체를 보내지 말고, 로컬 diff가 만든 “변경 후보 + 최소 컨텍스트(앞뒤 1~2문장, 표 일부, 이미지 해시)”만 전송  
섹션이 길면 “변경 후보 묶음” 단위로 프롬프트를 쪼개고, Claude는 “행(row) 생성”만 수행

#### Rate limit 및 재시도

Anthropic API rate limit 문서는 429 오류와 `retry-after` 헤더를 통해 재시도 대기 시간을 안내하며, 토큰 버킷 알고리즘을 사용한다고 설명합니다. citeturn3view0  
CLI에서도 내부적으로 API 요청을 수행할 수 있으므로, 병렬 실행 시 동시성 제한(예: 워커 수)과 지수 백오프를 적용하는 것이 안전합니다.

#### 대량 처리: Message Batches API(선택)

대량 요청을 처리해야 하고 즉시 응답이 필요 없을 때, Anthropic 문서는 Message Batches API가 “비동기 대량 처리”에 유용하며 *대부분 1시간 이내 완료* 및 *비용 50% 절감* 등을 언급합니다. citeturn5view0  
또한 배치 제한(요청 수/크기, 결과 보관 기간 등)과 모델 지원 범위가 문서로 제공됩니다. citeturn5view0  
단, Batch processing 문서에서 이 기능은 ZDR 미적용이라고 명시되어 있으므로, 민감 문서에서는 사용을 피하거나(또는 요약/마스킹 후) 조직 정책과 함께 판단해야 합니다. citeturn5view0turn10view2

### 보안·프라이버시

#### 데이터가 어디로 가는가

Claude Code Docs는 Claude Code가 로컬에서 실행되지만 LLM과 상호작용하기 위해 네트워크로 데이터를 보내며, 전송 데이터에는 “사용자 프롬프트와 모델 출력”이 포함된다고 명시합니다. citeturn10view0

비식별화/마스킹(정책 필요 시)

#### ZDR 선택지

Claude Code 권한 문서는 bypassPermissions 모드가 권한 프롬프트를 건너뛰며, “격리 환경에서만 사용”하라고 경고합니다. citeturn15view2

### 테스트·검증 전략

변경대비표 자동화는 “정확도”가 생명이라, 테스트가 필수입니다.

회귀 테스트 세트: 소형 DOCX(텍스트만/표만/그림만/혼합) + 기대 변경대비표(golden) 제작  
표/그림의 어려운 케이스: 병합 셀, 행/열 추가, 그림 교체/리사이즈/재압축  
섹션 분할 일관성 테스트: Heading 스타일이 깨진 문서에 대한 폴백 정책(예: “Heading이 없으면 N페이지 단위 chunk”)

`docxcompose`는 테스트 헬퍼/블랙박스 비교 방식(기대 docx fixture)도 언급하고 있어, “섹션 합치기” 기능 검증에 활용할 수 있습니다. citeturn16view0

LLM 단계는 `--json-schema`로 구조 검증을 강제할 수 있으므로 citeturn15view0turn4view0, 통합 테스트에서 “스키마 합격률/재시도율/빈 행 발생률” 같은 운영 지표를 함께 측정하는 것이 좋습니다.
